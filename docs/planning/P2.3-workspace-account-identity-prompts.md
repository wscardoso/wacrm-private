```
6. Verifique CNPJ: hoje só é setável NA CRIAÇÃO (via create_platform_workspace,
```

## FASE 3 — Plano de Implementação

```
P2.3 — Workspace Account Identity — Fase 3: Plano de Implementação

Com a Fase 1 e 2 validadas, proponha (ainda SEM IMPLEMENTAR) um plano de
implementação cobrindo:

1. Migration nova (04X_accounts_commercial_identity.sql ou nome equivalente):
   - legal_name (razão social) TEXT NULL
   - commercial_phone TEXT NULL
   - commercial_email TEXT NULL
   - Validação estrutural mínima (ex: formato de e-mail básico via CHECK ou
     validação em app-layer — justifique a escolha).
   - CNPJ: NÃO recriar a coluna (já existe desde 041); a migration só deve
     adicionar as 3 colunas novas.
2. RPC novo, espelhando o padrão de segurança de 042
   (create_platform_workspace): algo como
   update_platform_workspace_identity(p_account_id, p_name, p_legal_name,
   p_cnpj, p_commercial_phone, p_commercial_email).
   - SECURITY DEFINER, SET search_path = public.
   - REVOKE ALL FROM PUBLIC; GRANT EXECUTE TO authenticated.
   - Checagem de autorização: operador deve ter can_access_account(p_account_id)
     E ser 'admin' (não qualquer supervisor read-only) — justifique com base
     na resposta da Fase 2 item 3.
   - actor sempre auth.uid(), nunca argumento do caller.
   - Audit log: novo action 'update_workspace_identity' em platform_audit_log,
     no mesmo mecanismo de 042, dentro da mesma transação.
   - Todos os campos são opcionais/parciais (partial update) ou totais
     (full replace)? Decida e justifique — recomendo partial update (só
     sobrescreve o que for passado não-NULL) para não haver risco de um
     campo já preenchido virar NULL por omissão acidental no client.
3. UI nova: rota `/act/[accountId]/settings` (ou nome equivalente) —
   formulário read-then-write, reaproveitando requirePlatformContext() no
   layout já existente. Deve seguir o padrão de mutação já usado em outras
   telas platform (nenhuma — P2.1/P2.2 platform contacts é read-only; a
   referência de mutação real deve vir do lado MEMBER, ex:
   contact-detail-view.tsx, adaptado para o contexto platform).
4. Escopo explícito do que este lote NÃO faz (para não scope-creep):
   - Não mexe em ownership/onboarding do cliente (isso é P2.4).
   - Não adiciona validação de CNPJ com dígito verificador completo (fica
     como follow-up, igual ao já registrado para 041).
   - Não altera o fluxo member-side de rename (/api/account PATCH).

Entregável: plano escrito (sem código) com todos os pontos acima decididos e
justificados. Pare e aguarde aprovação explícita antes de escrever qualquer
arquivo.
```
