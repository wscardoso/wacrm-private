## 1. Objetivo e não objetivos

### 1.1 Objetivo

Definir o contrato arquitetural da **primeira capacidade que consome a credencial Meta** (fronteira decidida em `ADR-ATTR-002`) para **enriquecer** registros já capturados em `lead_attributions` (modelo congelado em `ADR-ATTR-001 §3.1`) com dados de campanha/conjunto/anúncio obtidos da **Graph API (Marketing)**, e para produzir um **relatório mínimo de attribution** a partir do resultado.

E6.0 realiza exatamente a fase **P1** descrita em `ADR-ATTR-001 §7` e o **D4** (§2: "capturar cru agora, enriquecer depois") — nada além disso.

### 1.2 Não objetivos (herdados do enunciado + reafirmados pela análise)

1. **CAPI / Conversions API (P3)** — retroalimentar Meta com eventos de conversão. Depende de sinal de venda (fora do domínio de attribution) — contrato próprio futuro (`ADR-ATTR-001 §0.3`).
2. **Envio de eventos para Meta** de qualquer natureza — E6.0 é **somente leitura** em relação à Graph API (consulta), nunca escrita.
3. **Otimização de campanha** — E6.0 não influencia veiculação, orçamento ou segmentação de anúncios.
4. **Fonte B (tracking links)** — `tracking_links`/`tracking_clicks` (P2) é épico separado (`ADR-ATTR-001 §0.3`); E6.0 não constrói nem depende dela.
5. **IA / scoring / automações comerciais** — fora do domínio de attribution; nenhuma inferência, nenhuma pontuação de lead, nenhum gatilho de automação é criado por E6.0.
6. **Alteração em messaging** — `messages`, `messages.status`, o pipeline de envio/entrega (E4a/E4b/DLB-001/ODI-001/ADR-MSG-001) permanecem intocados.
7. **Alteração em E4b** — o retry ledger, scheduler, orphan sweeper e classificador de falha de **entrega de mensagens** (`ARO-001`, `ADR-E4B-001/002/003`) não são tocados, estendidos, nem reusados como tabela/função. E6.0 **replica a disciplina** (classificação permanente/transitória, terminal único, backoff, TTL) como contrato **próprio**, independente — nunca a mesma instância.
8. **Alteração na captura de `ADR-ATTR-001`** — colunas de captura crua (`ad_source_*`, `ctwa_clid`, `origin_message_id`, `raw`, first-touch, `conversations.attribution_id`) são **somente leitura** para E6.0. Nenhuma migration de `033` é tocada.
9. **Alteração na fronteira de credencial de `ADR-ATTR-002`** — E6.0 **consome** a credencial pela fronteira já decidida (D-1…D-8); não redecide onde/como ela é armazenada, cifrada, provisionada, rotacionada ou revogada. Provisionamento/rotação/revogação continuam exclusivamente no RPC platform-side de `ADR-ATTR-002` (fora de E6.0).
10. **Fonte de verdade de campanha/anúncio** — E6.0 não cria um cadastro paralelo de campanhas; os dados de campanha/conjunto/anúncio são cache de enriquecimento, não uma entidade de domínio nova.

## 2. Fluxo arquitetural

```
[lead_attributions existente, capturado por ADR-ATTR-001]
        │  (ad_source_id presente, enriched_at IS NULL — seleção já
        │   nomeada em ADR-ATTR-001 §5; E6.0 não redecide o filtro)
        ▼
[Seleção + claim por tenant]  ── evita processamento concorrente duplicado
        │                          da mesma linha (disciplina equivalente
        │                          ao claim-as-lock de E4b — ledger PRÓPRIO,
        │                          não o de E4b)
        ▼
[Resolução da credencial do tenant]  ── via a fronteira ADR-ATTR-002
        │                                (D-7: só ciphertext sai do banco;
        │                                 decrypt só em processo de
        │                                 app/background)
        │
        ├── credencial inexistente/expirada/revogada → falha classificada
        │   como permanente PARA ESTE CICLO (§8/§9) — não chama a Graph API
        ▼
[Chamada à Graph API (Marketing)]  ── somente leitura (campanha/adset/
        │                              ad/placement/criativo)
        ▼
[Classificação do resultado]  ── permanente | transitória (§9)
        │
        ├── sucesso → persiste enriquecimento (colunas aditivas de
        │             ADR-ATTR-001 §3.1: campaign_id/name, adset_id/name,
        │             ad_id/name, placement, enriched_at) — NUNCA toca
        │             colunas de captura crua
        │
        ├── falha permanente → marca o ciclo de enriquecimento como
        │             encerrado para esta linha (terminal, não retry
        │             indefinido — disciplina de ADR-E4B-001 §6 aplicada
        │             ao domínio de enriquecimento)
        │
        └── falha transitória → volta elegível, sob backoff/TTL (§9)
        ▼
[Relatório mínimo de attribution]  ── leitura agregada de lead_attributions
                                       (enriquecido + não enriquecido) por
                                       tenant, via a RLS já existente
                                       (is_account_member, ADR-ATTR-001 §3.1)
```

**Camadas atravessadas, em ordem, e o que cada uma NÃO faz:**

1. Captura (`ADR-ATTR-001`, congelada) → não é reaberta; E6.0 só lê o que ela já escreveu.
2. Credencial (`ADR-ATTR-002`, aprovado) → não é reaberta; E6.0 só consome pela fronteira D-7/D-8.
3. Orquestração de enriquecimento (E6.0, este contrato) → nova, mas **não** é uma segunda instância do retry ledger de E4b nem do scheduler de E4b.
4. Relatório (E6.0, este contrato) → leitura pura, reusa autorização já existente, não introduz modelo novo.

## 3. Responsabilidades de cada camada

| Camada | Responsabilidade | Não-responsabilidade explícita |
|---|---|---|
| **Captura (ADR-ATTR-001)** | Já cumprida: grava `lead_attributions` cru, first-touch, ponteiro de conversa. | Não enriquece, não conhece Graph API, não conhece credencial. |
| **Credencial (ADR-ATTR-002)** | Já decidida: armazenamento, cifragem, provisionamento/rotação/revogação platform-side, estado (ativa/expirada/revogada). | Não decide *quando* ou *como* a credencial é consumida — isso é E6.0 (D-8 delega o invariante de consumo a E6.0, com teste obrigatório). |
| **Orquestrador de enriquecimento (E6.0)** | Selecionar linhas elegíveis por tenant; resolver e decifrar a credencial do tenant correto (nunca de outro); chamar a Graph API; classificar o resultado; persistir enriquecimento de forma aditiva e idempotente; conduzir retry/backoff/TTL sob disciplina própria; nunca persistir/logar plaintext do token. | Não decide autorização de quem PROVISIONA a credencial (isso é `ADR-ATTR-002 D-3`, já fechado). Não altera captura. Não escreve em `messages`/E4b. |
| **Relatório mínimo de attribution (E6.0)** | Ler `lead_attributions` (enriquecido + não enriquecido) por tenant e apresentar um resumo mínimo (contagem por fonte/campanha, taxa de enriquecimento, erros pendentes). | Não decide um novo modelo de autorização — herda a RLS já existente (`is_account_member(account_id)`, `ADR-ATTR-001 §3.1`). Não é um dashboard de BI, não faz IA/scoring. |
| **Auditoria (`platform_audit_log`, 037)** | Continua registrando exclusivamente **mutações privilegiadas de credencial** (`ad_credential_set`/`rotated`/`revoked`, já decidido em `ADR-ATTR-002 D-6`). | **Não** registra o *consumo* rotineiro da credencial pelo job de enriquecimento — ver §10 para o motivo e a alternativa contratual. |

## 4. Invariantes obrigatórios

1. **Aditividade sobre a captura.** Nenhuma escrita de E6.0 toca `ad_source_*`, `ctwa_clid`, `origin_message_id`, `raw`, `contacts.first_attribution_id`, `contacts.first_source_channel` ou `conversations.attribution_id`. Somente as colunas de enriquecimento já nomeadas em `ADR-ATTR-001 §3.1` (`campaign_*`, `adset_*`, `ad_*`, `placement`, `enriched_at`) são graváveis por E6.0.
2. **Nenhuma reabertura de `messages.status`.** E6.0 não lê nem escreve na máquina de estados de entrega. Nenhum acoplamento com `sending`/`sent`/`failed`.
3. **Nenhuma nova porta de autorização de sistema.** Se o job precisar rodar sem sessão humana (service_role), ele **reusa** a fronteira já decidida em `ADR-SYS-001` (fachada dedicada a `service_role`, sem checar `auth.uid()`, sem reabrir nenhuma RPC de `authenticated`) — não inventa um terceiro modelo de autorização.
4. **Plaintext do token nunca em repouso, nunca em log, nunca em `raw`/relatório.** Reafirmação direta de `ADR-ATTR-002 D-7` — E6.0 é exatamente o consumidor que aquela decisão antecipou; o invariante é herdado, não redecidido.
5. **Corte estrito por `account_id` (D-8).** A credencial usada para enriquecer uma linha de `lead_attributions` é **sempre e apenas** a do `account_id` daquela linha. Nenhum caminho de código pode, por partilhar um processo/lote/worker, aplicar a credencial de um tenant a linhas de outro.
6. **Terminal único por ciclo de enriquecimento.** Uma linha só é marcada como falha permanente **uma vez** por ciclo de tentativa (mesma disciplina de `ADR-E4B-001 §6`: terminal alcançado por decisão de desistência — classificação permanente, esgotamento de tentativas ou TTL — nunca como primeira reação a uma falha transitória).
7. **Idempotência de escrita.** Reprocessar a mesma linha com o mesmo resultado da Graph API não duplica dado nem produz efeito colateral observável distinto de uma única aplicação.
8. **Isolamento de falha por tenant.** A falha de credencial/Graph API de um tenant (expirada, revogada, rate-limited, permissão negada) nunca bloqueia, atrasa além do necessário, nem contamina o processamento de outro tenant.
9. **Sem escrita em `platform_audit_log` para consumo rotineiro.** Só mutação de credencial audita ali (já decidido). Ver §10 para o registro operacional próprio de E6.0.
10. **Sem nova tabela de identidade/segredo.** E6.0 não duplica nem reinterpreta a tabela de credenciais de `ADR-ATTR-002`; lê-a exclusivamente pela fronteira que aquele ADR definir na sua implementação.

## 5. Modelo de isolamento tenant

Duas camadas, herdadas e reafirmadas — nenhuma nova:

1. **Repouso (RLS).** `lead_attributions` já tem `SELECT` via `is_account_member(account_id)` (`ADR-ATTR-001 §3.1`); a credencial já tem RLS por `account_id` (`ADR-ATTR-002 D-5`). E6.0 não amplia nem estreita nenhuma das duas.
2. **Consumo (invariante contratual, não RLS).** Como o job roda tipicamente como `service_role` (fura RLS por natureza — mesmo raciocínio de `ADR-ATTR-002 §7.2`), o isolamento real vem do **invariante de join**: toda operação de enriquecimento de uma linha com `account_id = X` só pode resolver e usar a credencial cujo `account_id` também é `X`. Este é o mesmo invariante D-8 de `ADR-ATTR-002`, e **E6.0 herda a obrigação explícita de testá-lo** (teste cross-tenant, §11) — `ADR-ATTR-002 §7.2` já nomeia essa obrigação como pertencente a E6.0.
3. **Lote/worker nunca compartilha estado entre tenants.** Se o processamento é em lote (múltiplos tenants por execução), a resolução de credencial e a chamada à Graph API devem ocorrer em unidades isoladas por `account_id` — nenhuma variável, cache ou contexto de credencial pode "vazar" de uma iteração para a seguinte dentro do mesmo processo.

## 6. Uso correto da credencial

Regras herdadas de `ADR-ATTR-002` (D-2, D-4, D-7, D-8), aplicadas ao consumidor E6.0:

1. **Decrypt só em processo de aplicação/background**, nunca no banco (D-2/D-7). O job decifra em memória, usa o plaintext exclusivamente para autenticar a chamada à Graph API, e descarta — nunca grava o plaintext em `lead_attributions.raw`, em log, em métrica, em mensagem de erro, ou em qualquer linha de auditoria.
2. **Estado da credencial é checado antes do uso.** Se o estado (ativa/expirada/revogada — vocabulário de `ADR-ATTR-002 D-4`) não é "ativa", E6.0 **não tenta** a chamada à Graph API — trata como falha permanente para aquele ciclo (§4.6) e segue para a próxima linha/tenant.
3. **Um único uso por chamada.** A credencial é resolvida por linha (ou por lote-de-linhas-do-mesmo-tenant), nunca cacheada globalmente entre tenants diferentes na mesma execução (§5.3).
4. **Nenhuma escrita de volta na credencial pelo job.** E6.0 pode **sinalizar** que uma credencial parece inválida (ex.: a Graph API respondeu com erro de autenticação/permissão), mas **não** decide revogação — isso é ação de `ADR-ATTR-002 D-3` (platform-operator admin-para-o-tenant), fora de E6.0. O sinal, no máximo, é dado observável no relatório mínimo (§14).

## 7. Idempotência

1. **Seleção idempotente.** A query de elegibilidade (linhas com identificador de anúncio presente e ainda não enriquecidas) é a mesma nomeada em `ADR-ATTR-001 §5` — não redecidida aqui. Uma linha já enriquecida não é reprocessada automaticamente; um reprocessamento explícito (ex.: forçar novo enriquecimento) é decisão de produto fora deste contrato.
2. **Claim antes de processar.** Concorrência entre execuções do job (dois workers, ou um retry sobreposto a uma execução ainda em andamento) não pode enriquecer a mesma linha duas vezes nem produzir duas chamadas simultâneas à Graph API para a mesma linha. O mecanismo concreto (lock otimista, claim atômico, fila) é decisão de implementação — o contrato exige apenas que ele exista e seja **atômico ao nível de linha**, no mesmo espírito do padrão claim-as-lock já usado por E4b para o retry ledger de mensagens (referência de disciplina, não reuso de tabela).
3. **Escrita idempotente.** Persistir o mesmo resultado de enriquecimento duas vezes (ex.: por reprocessamento manual, ou por uma corrida que passou pelo claim) deve ser seguro — sobrescrever com o mesmo valor, não acumular, não duplicar linha.
4. **Nenhuma chave de idempotência nova é inventada para chamada externa.** A identidade natural da unidade de trabalho é a própria linha de `lead_attributions` (já com `id` único e `origin_message_id` único, herdados de `ADR-ATTR-001 §3.1`) — E6.0 não precisa (nem deve) criar um segundo identificador de idempotência para a mesma operação.

## 8. Tratamento de falhas externas

A Graph API (Marketing) é um sistema terceiro, com falhas de natureza distinta das falhas de entrega WhatsApp que E4b já trata — **por isso E6.0 não reusa o classificador de E4b**, mas replica sua disciplina:

1. **Falhas de autenticação/permissão** (token inválido/expirado/revogado, escopo insuficiente, objeto de anúncio não pertence mais à conta) → **permanentes** para aquele ciclo. Não retry automático — o problema está na credencial, não na chamada.
2. **Objeto inexistente/excluído** (campanha/adset/anúncio apagado no Meta Ads Manager) → **permanente**. A linha fica sem enriquecimento completo; isso é um resultado válido e final, não um erro a repetir.
3. **Rate limit (429 / erro de limitação da Graph API)** → **transitória**. Sujeita a backoff (§9), respeitando, quando disponível, o sinal de limitação retornado pela própria API.
4. **Erro de rede/timeout/5xx** → **transitória**. Mesma disciplina de backoff.
5. **Resposta malformada/inesperada** → tratada como **transitória** por padrão de segurança (mesmo princípio de `DLB-001 §10.1`: o custo de subestimar uma falha recuperável é menor que o de presumir uma falha permanente sem certeza) — mas com um teto de tentativas mais conservador, para não mascarar um bug de parsing como "rate limit eterno".
6. **Falha de credencial não-ativa (§6.2)** → tratada como permanente **sem sequer chamar a Graph API** — é uma pré-condição, não uma resposta da API.

Em nenhum caso uma falha de enriquecimento pode:
- reverter ou corromper dado de captura já persistido (`ADR-ATTR-001`);
- propagar-se para o caminho de mensageria (E4a/E4b);
- bloquear a captura de novas atribuições (a captura, por D4, é sempre imediata e independente do enriquecimento).

## 9. Retry policy (contrato — não implementação)

1. **Classificação antes de contar tentativa** — mesmo princípio de `ARO-001`/`ADR-E4B-001` aplicado ao domínio de enriquecimento: uma falha permanente não consome uma "tentativa" de retry, porque não haverá próxima tentativa.
2. **Falha transitória → backoff com teto de tentativas e TTL.** Os valores concretos (número de tentativas, curva de backoff, janela de TTL) são decisão de implementação, não deste contrato — o contrato exige apenas que **ambos existam** (teto finito de tentativas **e** expiração temporal), para que uma linha nunca fique indefinidamente "em processamento".
3. **Esgotamento de teto ou TTL → terminal permanente para o ciclo**, exatamente como `ADR-E4B-001 §6` define para `failed`: alcançado uma única vez, como decisão de desistência, nunca revertido silenciosamente. Um novo ciclo de enriquecimento (se o produto decidir permitir refresh) é uma operação **explícita e distinta**, fora deste contrato.
4. **Retry é por linha, não por tenant.** A falha (mesmo permanente) de uma linha não marca o tenant inteiro como inelegível — outras linhas do mesmo tenant continuam sendo processadas normalmente, exceto no caso de §8.6 (credencial não-ativa), que é uma condição do tenant como um todo e, portanto, afeta todas as suas linhas elegíveis até ser resolvida (fora de E6.0, via `ADR-ATTR-002 D-3`).
5. **Nenhum retry cruza o limite de `ADR-ATTR-002`.** Se a causa raiz é a credencial (expirada/revogada), retry não resolve o problema — o contrato exige que a falha seja **observável** (relatório mínimo, §14) para que a ação corretiva (reprovisionamento, fora de E6.0) seja tomada por quem tem autoridade sobre ela.
6. **Retry ledger de E6.0 é próprio.** Não é o retry ledger de mensagens de E4b (`ARO-001`), nem uma extensão dele. Estrutura, nome e localização são decisão de implementação — fora deste contrato.

## 10. Auditoria necessária

1. **Mutação de credencial** (provisionar/rotacionar/revogar) já está coberta por `ADR-ATTR-002 D-6` — `platform_audit_log`, mesma transação, ator + tenant + ação, nunca o valor. **E6.0 não adiciona nem modifica isso.**
2. **Consumo rotineiro da credencial por E6.0 NÃO é gravado em `platform_audit_log`.** Justificativa: aquela tabela é reservada a **mutações privilegiadas de identidade/credencial** com ator humano (`actor_user_id` obrigatório, `037`) — um job de sistema, sem `auth.uid()`, não tem ator humano a registrar, e registrar uma execução de rotina (potencialmente centenas de linhas por tenant) ali degradaria o propósito e o volume daquele log, mesmo raciocínio de escopo que `E5 §9` já aplicou para não colocar mutação self-service de tenant em `platform_audit_log`.
3. **O rastro operacional de E6.0 vive nos próprios dados que ele produz** — mesmo princípio de `E5a`: `lead_attributions.enriched_at` (quando um enriquecimento teve sucesso) e o estado de falha/retry de uma linha (quando não teve) já constituem, por si, um registro de "o que E6.0 fez e quando", sem necessidade de tabela de auditoria dedicada dentro deste contrato.
4. **Exceção de auditoria obrigatória: falha permanente atribuível à credencial.** Quando E6.0 conclui que uma credencial está inutilizável (expirada, revogada, ou rejeitada pela Graph API por autenticação/permissão — §8.1/§6.2), esse fato deve ser **observável** por quem tem autoridade para agir (o operador de plataforma que provisiona a credencial, `ADR-ATTR-002 D-3`) — através do relatório mínimo (§14) ou de um sinal operacional equivalente, nunca silenciosamente engolido pelo classificador de retry.
5. **Nenhum valor de segredo, em nenhuma hipótese, aparece em qualquer registro produzido por E6.0** — reafirmação de `ADR-ATTR-002 D-7` (§6.1 acima), testável (§11).

## 11. Testes obrigatórios

1. **Isolamento cross-tenant (D-8, crítico).** Dado dois tenants com credenciais distintas, processar linhas de ambos no mesmo lote/execução e provar que a credencial de A nunca é usada para enriquecer uma linha de B (e vice-versa) — mesmo sob execução concorrente.
2. **Nenhum vazamento de plaintext.** Inspecionar toda saída observável de uma execução (linha persistida, relatório, erro, log estruturado do teste) e provar ausência do valor plaintext do token em qualquer uma delas.
3. **Idempotência de escrita.** Processar a mesma linha duas vezes (mesmo resultado de Graph API simulado) e provar que o efeito final é idêntico a uma única aplicação — sem duplicação, sem erro.
4. **Claim atômico sob concorrência.** Duas execuções concorrentes disputando a mesma linha elegível — exatamente uma processa; a outra não duplica a chamada externa nem a escrita.
5. **Classificação de falha — permanente.** Autenticação/permissão negada, objeto inexistente → linha marcada terminal, sem re-tentativa automática.
6. **Classificação de falha — transitória.** Rate limit / timeout / 5xx simulado → linha permanece elegível para retry sob backoff, até teto/TTL.
7. **Credencial não-ativa.** Estado expirado/revogado → job não chama a Graph API; falha classificada como permanente sem tentativa externa (§6.2/§8.6).
8. **Isolamento de falha por tenant.** Falha (mesmo total) no processamento do tenant A não impede nem atrasa indevidamente o processamento do tenant B na mesma execução.
9. **Aditividade sobre a captura.** Após um ciclo de enriquecimento (sucesso ou falha), toda coluna de captura crua de `ADR-ATTR-001` permanece byte-a-byte idêntica ao valor pré-enriquecimento.
10. **Regressão de fronteira.** `messages`, `messages.status`, tabelas/RPCs de E4b, e a fronteira de armazenamento/autorização de `ADR-ATTR-002` permanecem intocadas por qualquer teste de E6.0 (nenhuma migration de mensageria ou de credencial é alterada por E6.0).
11. **Relatório mínimo — corretude.** Para um conjunto conhecido de linhas (parte enriquecida, parte não, parte com falha permanente registrada), o relatório reflete exatamente essas contagens/estado por tenant, restrito ao tenant do chamador (via a RLS já existente).
12. **Relatório mínimo — isolamento de leitura.** Um membro do tenant A não enxerga, no relatório, nenhuma linha do tenant B (reforço do teste de RLS já existente em `ADR-ATTR-001`, aplicado à nova superfície de leitura).

## 15. Observabilidade operacional do ciclo de enriquecimento

### 15.1 Objetivo

Garantir que o operador do sistema (humano ou automatizado) consiga responder, para cada linha de `lead_attributions` elegível para E6.0, as seguintes perguntas sem depender de depuração em produção, rastreamento de log disperso ou acesso direto ao banco:

- qual linha foi processada (identificador);
- quando a última tentativa ocorreu;
- quantas tentativas já foram realizadas;
- qual a classificação da última falha (permanente/transitória/bloqueada);
- qual o último erro operacional (código/mensagem segura, sem segredo);
- qual o estado atual do enriquecimento (pendente, em andamento, concluído, falha permanente, expirado).

### 15.2 Responsabilidades

| Camada | Responsabilidade |
|---|---|
| **Job de enriquecimento (E6.0)** | Persiste, na mesma transação do processamento da linha, os metadados que permitam responder às perguntas de §15.1 — estado do ciclo, tentativas, `last_attempt_at`, classificação de falha, erro operacional. |
| **Retry ledger de E6.0 (tabela própria)** | Porta os metadados de tentativa por linha: `attempt_count`, `last_attempt_at`, `last_error`, `last_outcome_class`, `status` (elegível/claimado/concluído/falha-permanente). |
| **Colunas de enriquecimento em `lead_attributions`** | Já portam `enriched_at` (implicitamente responde "concluído com sucesso"). Demais metadados de tentativa (tentativas, erros, estado) **não** são gravados em `lead_attributions` — vivem exclusivamente no ledger próprio. |
| **Relatório mínimo (§14)** | Expõe, na resposta autorizada por tenant, o estado de cada linha (enriquecida, pendente, falha permanente) de forma agregada ou individual — consumindo a leitura do ledger próprio, nunca de `platform_audit_log`. |

### 15.3 Invariantes

1. **Observabilidade como subproduto do processamento, não como sistema separado.** Os metadados de §15.2 são escritos na **mesma transação** que o claim/resultado da tentativa — nunca em lote diferido, nunca em tabela de auditoria separada. Se a transação falha, não há metadado órfão.
2. **Estado sempre rastreável.** Uma linha elegível está sempre em um dos estados: `pending` (nunca processada), `claimed` (em processamento), `completed` (enriquecida), `failed_permanent` (ciclo encerrado por falha permanente, esgotamento ou TTL), `expired` (TTL venceu sem conclusão). Nenhuma linha pode estar "invisível" entre o claim e a conclusão.
3. **Toda transição registra o ator do sistema, não um usuário humano.** O job roda como `service_role` (ou equivalente, §4.3); o "ator" registrado é um identificador fixo do job (ex.: `'enrichment-job'`), nunca `auth.uid()`, nunca um ID de usuário real — porque não há sessão humana.
4. **Erro operacional sem segredo.** O campo `last_error` do ledger contém um código de erro normalizado (ex.: `'graph_api_auth_error'`, `'rate_limited'`, `'network_timeout'`, `'object_not_found'`) ou, quando o erro original não classificar em nenhuma categoria conhecida, um placeholder `'unclassified'` — nunca o raw response body da Graph API, nunca o valor do token, nunca um stack trace completo.
5. **Nunca reescrever o estado de captura.** Nenhum metadado de observabilidade de E6.0 (tentativas, erros, estado de ciclo) é escrito em colunas de `lead_attributions` definidas em `ADR-ATTR-001 §3.1` — essas colunas são somente leitura para E6.0. O ledger próprio é a única fonte de observabilidade do ciclo.

### 15.4 Informações permitidas no ledger próprio de E6.0

| Campo | Origem |
|---|---|
| `message_id` (PK, FK para `lead_attributions.id`) | Identificador da linha processada |
| `status` | `'pending'`, `'claimed'`, `'completed'`, `'failed_permanent'`, `'expired'` |
| `attempt_count` | `INTEGER NOT NULL DEFAULT 0` |
| `last_attempt_at` | `TIMESTAMPTZ` |
| `last_outcome_class` | `'permanent'`, `'transient'`, `'blocked'` |
| `last_error` | `TEXT` — código normalizado (ver §15.3.4) |
| `locked_until` | `TIMESTAMPTZ` — claim-as-lock |
| `ttl_expires_at` | `TIMESTAMPTZ` — horizonte de expiração |
| `created_at`, `updated_at` | Carimbo de criação/atualização |

Nenhum destes campos pode conter, em nenhuma hipótese, o valor plaintext da credencial, o response body bruto da Graph API, ou qualquer dado de `lead_attributions` já capturado (que permanece em sua coluna original e não é duplicado aqui).

### 15.5 Informações proibidas em qualquer registro de E6.0

1. **Plaintext do token de acesso** (reafirmação de `ADR-ATTR-002 D-7` e §4.4 deste contrato).
2. **Response body bruto da Graph API** — o ledger só recebe o código normalizado (`last_error`), nunca o payload inteiro.
3. **Dados de captura de `ADR-ATTR-001`** duplicados — o ledger referencia por FK (`message_id`), não copia `ad_source_id`, `ctwa_clid`, `raw` ou qualquer coluna da captura.
4. **Identificador de usuário humano** (`auth.uid()`) — o ator do job é um identificador fixo de sistema (§15.3.3).
5. **Stack trace completo** ou mensagem de erro interna não normalizada.

### 15.6 Critérios de teste específicos para observabilidade

1. **Transição de estado visível.** Para cada estado do ciclo (`pending → claimed → completed` ou `pending → claimed → failed_permanent`), o ledger reflete o estado corrente após cada operação — verificado com leitura direta do ledger após o processamento simulado.
2. **Nenhum estado invisível.** Após um claim sem conclusão (simulação de crash/morte do worker antes da escrita do resultado), a linha permanece em `claimed` com `locked_until` no futuro — o orphan sweeper contratual de E6.0 (ou mecanismo equivalente) deve reciclá-la; o teste verifica que o ledger não deixou a linha sem estado rastreável.
3. **Erro normalizado, não bruto.** Após uma falha simulada da Graph API (ex.: rate limit, auth error), o campo `last_error` contém `'rate_limited'` ou `'graph_api_auth_error'` ou similar — nunca o JSON de resposta crua nem o stack trace do erro HTTP.
4. **Nenhum escrito em `lead_attributions`.** Após qualquer cenário (sucesso, falha, expiração), as colunas de `ADR-ATTR-001 §3.1` em `lead_attributions` permanecem inalteradas — exceto `enriched_at` exclusivamente no caso de sucesso, conforme já contratado em §4.1.
5. **Ator de sistema, não `auth.uid()`.** O ledger de E6.0 (ou metadado equivalente) nunca referencia `auth.uid()` — verificado por teste de schema ou por assinatura de função.
6. **Relatório mínimo reflete estado observável.** Para uma combinação conhecida de linhas (uma concluída, uma em `pending`, uma em `failed_permanent`), o relatório mínimo retorna exatamente esses estados por linha, no escopo do tenant autorizado.
